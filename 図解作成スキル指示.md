# 図解作成スキルへの指示内容

## ツールの概要（タイトル・説明）

**ツール名：新店舗オープン管理ツール**

酒類営業担当者が、複数の担当店舗の新規オープン準備を一元管理するための業務支援ツール。画面は左から右へ4ペイン構成で、店舗を選ぶと右側の詳細が連動して表示される。

---

## 1. ツールの画面のキャプチャ

画面は以下の4ペイン構成：

| ペイン | 内容 |
|---|---|
| Pane 1（左サイドバー） | 店舗一覧。「進行中」「未開始」「完了」のグループにドラッグ＆ドロップで振り分け |
| Pane 2 | 選択した店舗のプロフィール（住所・担当者・納品条件など約40項目） |
| Pane 3 | その店舗のタスク一覧。信号機マーク（🔴🟡🟢）で期日の切迫度を表示 |
| Pane 4 | タスクの詳細・メモ・期日の編集フォーム |

---

## 2. 保持できるようにしたデータ

Vercel Postgresに以下の4テーブルで保存：

**① stores（店舗テーブル）**
- 店舗ID、進捗ステータス（未開始 / 進行中 / 完了）

**② store_profiles（店舗プロフィールテーブル）**
- 得意先コード、店名、法人名、業態
- 住所、電話番号、担当者名
- 支払方法、回収担当者
- 納品時間帯（開始・終了）、定休日
- 初回納品日、オープン日
- 喫煙ポリシー、座席数、客単価、予想売上
- WEB注文・協賛・新規店・雑ボトル管理・鍵預かり・花束・代理納品など ON/OFFフラグ 7種
- メモ・備考・添付メモ など合計約40項目

**③ tasks（タスクテーブル）**
- タスクID、店舗ID、タスク名
- 種別（通常 / 号車報告書）
- ステータス（未着手 / 進行中 / 完了 / N/A）
- 期日、メモ
- プロフィール連動フラグ（例：「WEB注文あり」のときのみ表示するタスク）

**④ subtasks（子タスクテーブル）**
- 子タスクID、親タスクID、名前
- 完了フラグ（true/false）
- 雑ボトル管理連動フラグ、下部固定フラグ

---

## 3. 保存先と、それを選んだ理由

**保存先：Vercel Postgres（クラウドSQL データベース）**

| 理由 | 詳細 |
|---|---|
| サーバー管理が不要 | Vercel上でデータベースを数クリックで作成・接続でき、インフラの知識がなくても動く |
| Next.jsとの相性が抜群 | `@vercel/postgres` のSQL関数を使うだけで、Next.jsのServer Actionsから直接読み書きできる |
| ページをリロードしてもデータが残る | ブラウザを閉じても、別のデバイスで開いても同じデータが見える「永続化」が実現できる |
| 学習・開発用の無料枠がある | スクールの課題制作レベルであれば費用がかからない |

比較：ブラウザのlocalStorageに保存する方法もあったが、別デバイスや別ブラウザでは見えなくなるため不採用。

## 4. 工夫したポイントと苦戦したポイント

### 工夫したポイント

| ポイント | 内容 |
|---|---|
| 信号機ロジック | タスクの期日と今日の日付を自動比較し、🔴🟡🟢で切迫度を色表示。毎分チェックして日付をまたいでも自動更新 |
| プロフィール連動タスク | 店舗プロフィールの「WEB注文あり」「協賛あり」などのON/OFFに連動して、関連タスクが自動で表示・非表示になる |
| 自動完了ルール | 全タスクが完了した店舗は、ドラッグ操作なしで自動的に「完了」グループに移動する |
| 4ペイン連動 | 店舗を選ぶ→タスク一覧が更新→タスクを選ぶ→詳細が表示、という自然な操作の流れ |

### 苦戦したポイント

| ポイント | 内容 |
|---|---|
| DBテーブル設計 | 約40項目のプロフィールをどのカラムに入れるか、日本語フィールド名を英語スネークケースに変換する作業が大変だった |
| boolean値の保存 | VercelPostgresがtrue/falseをそのまま保存できないため、`'true'`/`'false'`の文字列に変換して保存・復元する処理が必要だった |
| 条件付きタスクの管理 | プロフィールのフラグ変更が即座にタスク表示に反映されるよう、UIの状態管理を整理するのに時間がかかった |
| Server Actionsの設計 | Next.jsのサーバー側関数（Server Actions）でDBを操作する仕組みを理解し、ページとコンポーネントをつなぐ構造を把握するまでが難しかった |
